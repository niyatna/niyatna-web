The **Context Explorer** helps Niyatna attach relevant company knowledge to a request before it is routed.

It should answer: what does the system need to know to complete this work safely and correctly?

---

## Context layers

Niyatna can attach several kinds of context:

* **Company canon:** Mission, standards, product definitions, and operating rules.
* **Department context:** SOPs, responsibilities, checklists, and recurring workflows.
* **Source material:** Files, records, docs, links, screenshots, or submitted outputs.
* **Decision history:** Prior approvals, rejected paths, and known constraints.
* **Proof history:** Evidence from similar completed or blocked work.
* **Preference memory:** Brand, design, copy, security, and workflow preferences.

## Rule: intent priority

Context supports the active intent. It does not replace it.

If old context conflicts with a current authorized instruction, the current instruction wins. Niyatna should inject only the context needed for the active goal so work stays focused instead of drifting into old assumptions.
